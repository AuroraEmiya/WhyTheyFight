# WhyTheyFight 开发者设计文档

本文件用于记录 WhyTheyFight 当前阶段的游戏设计共识，供开发者、策划和协作者阅读。

---

# 1. 游戏概述

## 1.1 核心玩法定位

《WhyTheyFight》融合三类玩法：

```text
1. 沙盘战争
2. 回合制战斗
3. RTS 资源运营与单位调度
```

游戏目标是在保留较高自由度和策略深度的同时，避免传统 RTS 对操作速度、反应速度、多线操作和高频微操的强依赖。

玩家主要通过：

```text
地图判断
资源规划
单位调度
战术执行
防御与进攻时机选择
胜利路线选择
```

取得优势。

---

## 1.2 沙盘战争

游戏希望通过随机地图生成机制，使每一局都产生不同的战略环境。

随机生成内容包括：

```text
地图整体结构
地形分布
资源点位置
玩家出生区域 spawn_space
关键战略位置
玩家之间的初始距离和接触路径
```

`spawn_space` 是玩家开局阶段拥有或可使用的初始部署区域。它是玩家建立核心、部署单位和展开资源运营的起点。

地图随机性的目的不是单纯制造混乱，而是让每局产生不同的战略判断。

---

## 1.3 回合制战斗

本游戏采用回合制或类回合制行动结构，而不是传统即时 RTS 的实时操作结构。

回合制结构的目标：

```text
1. 让玩家有时间思考和规划；
2. 让关键决策更加清晰；
3. 降低手速对胜负的影响；
4. 让战斗结果更容易理解和复盘；
5. 控制单局时长，避免游戏节奏过慢。
```

---

## 1.4 RTS 元素

游戏保留 RTS 中最核心的两个系统：

```text
1. 战斗单位的生产、部署、移动、进攻和防御；
2. 资源的采集、运营、扩张和转化。
```

玩家不仅需要考虑局部战斗，也需要考虑长期运营。

---

# 2. 单局目标

## 2.1 基础胜利：核心摧毁胜利

每个玩家拥有一个核心对象：

```text
core
```

基础胜利规则：

```text
当某个玩家摧毁场上所有敌对玩家的 core 后，该玩家获得胜利。
```

当前基础版本优先实现单人阵营对抗，不优先处理复杂同盟胜利。

---

## 2.2 飞升胜利：发育型胜利

飞升胜利是后续版本计划开发的高级胜利方式。

基本概念：

```text
某个玩家通过长期发育、资源积累、科技推进或特殊成就，达成“机械飞升”条件，从而获得胜利。
```

飞升胜利当前属于待开发内容，只作为系统预留方向存在。

---

# 3. 游戏时长与节奏

单局目标时长：

```text
30 分钟 ~ 1 小时
```

理想节奏：

```text
前期：
- 探索
- 部署
- 采集
- 判断地图
- 选择初步发展方向

中期：
- 资源争夺
- 防御建设
- 小规模战斗
- 扩张与反扩张

后期：
- 核心进攻
- 飞升压力
- 资源优势转化为终结能力
- 优势方推动游戏结束
```

---

# 4. 玩家人数与阵营

游戏支持至少 2 名玩家。

```text
player_count >= 2
```

默认情况下：

```text
阵营数量 = 玩家人数
```

基础版本可以先采用：

```text
所有非己方玩家默认互相敌对。
```

后续可以扩展为：

```text
临时结盟
合作
背叛
共享视野
互不攻击协议
共同胜利
资源贸易关系
```

# 5. 地图系统

## 5.1 地图大小

地图暂定大小：

```text
100 * 100
```

地图由网格组成，每个格子使用坐标表示：

```text
Vector2i(x, y)
```

---

## 5.2 地图数据分层

地图系统不能只用一个“格子类型”表示所有对象。

推荐地图层级：

```text
Terrain Layer    地形层
Structure Layer  建筑层
Unit Layer       单位层
Resource Layer   资源层
Vision Layer     视野层
Effect Layer     效果层
```

一个格子可以同时具有多层信息。

---

## 5.3 地形类型

当前基础地形包括：

```text
Road
Water
Obstacle
```

### Road

```text
所有可移动单位默认可以通行。
可以放置允许陆地部署的建筑。
建筑放置后，该格子被建筑占用。
```

### Water

```text
只有具有 can_swim 标记的单位可以通行。
只有具有 can_build_on_water 标记的建筑可以放置。
```

### Obstacle

```text
默认不可通行。
默认不可建造。
可以作为山体、废墟、阻挡物等。
```

---

## 5.4 建筑 / 不可移动结构

当前建筑类型：

```text
Core
Tower
AttackTower
DetectTower
```

### Core

用途：

```text
作为玩家的关键目标。
被摧毁后可能导致玩家失败或出局。
```

待定：

```text
Core 是否可以生产单位？
Core 是否可以攻击？
Core 是否可以提供资源？
Core 是否可以移动？
Core 被摧毁后玩家是否立刻出局？
```

### AttackTower

用途：

```text
对攻击范围内的敌对单位或建筑造成伤害。
```

### DetectTower

用途：

```text
提供一定范围的视野。
可能用于发现敌方单位、资源点或隐形目标。
```

---

## 5.5 可移动单位

当前单位类型：

```text
EngineerUnit
ForceUnit
```

### EngineerUnit

可能职责：

```text
采集资源
建造建筑
修复建筑
部署设施
```

### ForceUnit

可能职责：

```text
攻击敌方单位
攻击敌方建筑
防守核心
争夺资源点
```

---

## 5.6 资源点

当前资源方向包括：

```text
矿石
食物
可激活单位
水源
```

### 矿石

```text
前期：
- 铁矿

后期：
- 铜矿
```

### 食物

```text
野外食物
作物
```

### 可激活单位

示例：

```text
Magic
...
```

### 水源

```text
未污染水源
已污染水源
```

---

## 5.7 可通行规则

基础规则：

```text
Road：
- 默认允许所有可移动单位通行。
- 放置建筑后，该格子被占用，普通单位不可通行。

Water：
- 只有 can_swim 单位可以通行。
- 只有 can_build_on_water 建筑可以放置。

Obstacle：
- 默认不可通行。
- 默认不可建造。
```

推荐判断顺序：

```text
1. 目标格是否存在；
2. 地形是否允许该单位进入；
3. 是否存在阻挡建筑；
4. 是否存在阻挡单位；
5. 是否存在特殊状态或效果；
6. 返回最终可通行结果。
```

---

## 5.8 视野规则

视野来源：

```text
某些塔
某些可移动单位
某些能力或技能
可能存在的地图资源或特殊建筑
```

基础设计：

```text
每个玩家拥有独立视野。
单位或建筑提供 vision_range。
处于 vision_range 内的格子对该玩家可见。
```

# 6. 单位系统

## 6.1 单位基础属性

```text
HP   Health Point，生命值
SP   Skill Point，技能点
PDP  Physical Defense Point，物理防御
MDP  Magical Defense Point，魔法防御
PAP  Physical Attack Point，物理攻击
MAP  Magical Attack Point，魔法攻击
AR   Attack Range，攻击范围
MOV  Mobility per turn，每回合移动力
Relic 遗物 / 特性 / 状态附着物
```

---

## 6.2 单位能力标记

推荐使用能力标记，而不是只靠大类判断：

```text
can_move
can_attack
can_gather
can_construct
can_defend
can_use_skill
can_swim
can_provide_vision
```

---

## 6.3 单位状态

当前状态包括：

```text
Alive
Dead
Stunned
Debuffed
StatusUp
```

基础规则：

```text
Dead 单位不能行动。
Stunned 单位不能执行主动行动。
Debuffed 单位根据具体 debuff 修改属性或行动结果。
StatusUp 单位根据具体 buff 提升属性或行动结果。
```

---

# 7. 行动系统

## 7.1 行动点 / 指令

每个玩家拥有初始行动点：

```text
action_points = x
```

当前基础规则：

```text
1. 每个玩家每回合拥有一定数量行动点；
2. 部署单位需要消耗行动点；
3. 建造建筑需要消耗行动点；
4. 对单位执行行动可能消耗行动点；
5. 单位在行动当轮可以接受单个主要指令。
```

建议基础版本先采用：

```text
每个单位每回合最多执行一个主要行动。
玩家的行动点限制本回合总操作数量。
```

---

## 7.2 行为类型

```text
MOVE(fromSrc, toDst)
ATTACK(fromSrc, toDst)
GATHER(fromSrc, toDst)
CONSTRUCT(fromSrc, toDst, structure_type)
DEFEND(fromSrc)
SKILL(fromSrc, toDst, skill_id)
```

# 8. 战斗系统

## 8.1 攻击范围

攻击范围可以参考《明日方舟》的矩阵式攻击范围。

示例：

```text
近战范围：
[0, 1, 0]
[1, C, 1]
[0, 1, 0]
```

其中 `C` 表示攻击者所在格。

---

## 8.2 伤害计算

当前基础公式方向：

```text
Damage = BaseDamage * AttackModifier * VulnerabilityModifier * DefenseReduction
```

原型期可先采用简单公式：

```text
FinalDamage = max(1, AttackPoint - DefensePoint)
```

后续再切换到更平滑的乘区公式。

---

## 8.3 反击 / 先后手

当前未定义。

基础版本建议：

```text
暂不实现反击。
攻击方先结算伤害。
目标 HP <= 0 时立即死亡或摧毁。
```

---

## 8.4 随机性

随机性原则：

```text
高风险，高回报。
```

随机性主要用于：

```text
暴击
闪避
特殊技能
遗物效果
高风险资源点
地图事件
可激活单位
```

不建议大量用于：

```text
普通移动
普通采集
普通建造
基础攻击是否命中
```

---

## 8.5 死亡与摧毁

建议基础规则：

```text
当单位或建筑 HP <= 0 时，进入 Dead / Destroyed 状态。
Dead 单位不能行动。
Destroyed 建筑不再提供功能。
死亡单位从 Unit Layer 移除。
被摧毁建筑从 Structure Layer 移除，或保留为 Ruin。
```

Core 特殊规则：

```text
当 Core 被摧毁时，触发 core_destroyed 事件。
VictorySystem 重新检查胜利条件。
```

---

# 9. 后续优先级

当前最应该优先明确的是回合系统。

建议下一步优先定义：

```text
1. 回合结构
2. 行动阶段
3. 行动点恢复
4. 单位行动限制
5. 回合结束条件
```

最关键问题：

```text
一回合中，玩家到底能做多少事？
```
